# 开发问题复盘：小程序头像上传后不显示

日期：2026-05-31

## 一、问题现象

在微信开发者工具真机调试中，服务人员进入「编辑个人资料」上传头像后：

- 当前编辑页面可立即看到刚选择的头像；
- 点击保存后返回首页，首页头像不显示；
- 再次进入个人资料编辑页，头像也不显示；
- 后端日志显示上传、保存、读取接口均返回成功；
- 后端没有 403/404 报错。

相关后端接口日志表现：

```text
POST /api/app/files/upload 201
PUT /api/app/profile 200
GET /api/app/profile 200
```

## 二、结论

这不是单纯的 Claude 编码问题，也不是单纯的 Codex 命令脚本问题。

更准确地说，是三个问题叠加：

1. Claude 的实现偏向“静态代码验证”，没有覆盖微信真机图片加载场景。
2. Codex 前几轮复审偏向“字段保存链路”，没有第一时间用接口冒烟测试拆分保存问题和图片预览问题。
3. 本地开发环境使用 `http://局域网IP:3000/api/app/files/public/{fileId}/preview` 作为头像图片地址，真机小程序的 `<image>` 组件在该环境下没有稳定加载这个本地 HTTP 预览地址。

最终确认：后端实际已经能保存头像 `fileId`，也能通过公开预览接口返回图片；问题主要发生在真机小程序端图片显示链路。

## 三、为什么前面多轮没有解决

### 1. 验证方式偏静态

前面多次让 Claude 自审，主要通过：

- TypeScript build；
- 小程序 JS 语法检查；
- 关键字符串 marker 检查；
- 局部逻辑检查。

这些验证能发现编译错误和明显逻辑缺失，但不能证明微信真机环境里的 `<image>` 是否能加载本地 HTTP 图片。

### 2. 把“头像不显示”误判成“头像没保存”

一开始的修复方向主要围绕：

- `avatarUrl` 是否写入 `StaffProfile`；
- 是否同步到 `StaffAccount.wechatAvatar`；
- 保存后是否重新拉取 `/api/app/profile`；
- 前端多个页面是否统一使用头像字段。

这些修复本身是有价值的，但还没有把问题拆成两段：

```text
上传/保存/读取是否成功
图片组件是否真的加载图片 URL
```

直到后面通过接口冒烟测试确认：

```text
fileId 有值
updatedAvatarUrl 有值
GET /profile 返回 avatarUrl
/files/public/{fileId}/preview 返回 200 image/png
```

才明确后端链路是通的。

### 3. 缺少运行期诊断日志

早期没有在小程序端输出以下关键数据：

- 上传接口返回的 `fileId`；
- 保存接口返回的 `avatarUrl`；
- 重新读取 profile 后的 `profile.avatarUrl`；
- 页面最终用于 `<image>` 的 `src`；
- `<image>` 的 `binderror` 事件。

导致排查依赖现象描述，定位效率低。

### 4. 本地真机调试环境和正式环境不同

正式上线后，小程序图片应走 HTTPS 合法域名或 OSS/CDN。

但本地真机调试使用的是局域网 HTTP 地址，例如：

```text
http://192.168.x.x:3000/api/app/files/public/{fileId}/preview
```

接口本身能请求成功，不代表 `<image>` 组件一定会稳定加载该图片地址。这个差异是前面没有充分考虑的。

## 四、最终修复方案

### 1. 后端增强

文件：

- `apps/server/src/modules/staff/staff.service.ts`

处理：

- 保存头像时写入 `StaffProfile.avatarUrl`；
- 同步写入 `StaffAccount.wechatAvatar` 作为兜底；
- 读取个人资料时，`profile.avatarUrl` 和 `account.wechatAvatar` 互为兜底；
- 返回 `avatarUrl` 和 `avatarFileId`，减少前端字段歧义；
- 将头像文件提升为 public，保证公开预览接口可访问。

### 2. 小程序头像工具统一

文件：

- `apps/miniapp/utils/avatar.js`

处理：

- 统一解析上传响应中的 `fileId`；
- 统一将 `fileId` 转为预览地址；
- 增加 `resolveAvatarValue()`，统一从 profile、account、本地缓存中解析头像；
- 增加 `staffAvatarFileId` 缓存；
- 增加 `staffAvatarLocalPreview` 缓存，用于本地真机调试兜底。

### 3. 页面统一使用头像解析工具

涉及页面：

- `apps/miniapp/pages/profile/edit/index.js`
- `apps/miniapp/pages/profile/view/index.js`
- `apps/miniapp/pages/home/index.js`
- `apps/miniapp/pages/account/index.js`
- `apps/miniapp/pages/resume/index.js`

处理：

- 所有页面统一使用 `resolveAvatarValue()` 和 `normalizeAvatarUrl()`；
- 不再各自拼接或判断头像地址；
- 上传后保存 `fileId`，同时缓存手机本地临时图片路径；
- 本地真机调试阶段优先使用本地临时路径显示头像。

## 五、当前方案的性质

当前方案是“开发期稳定兜底 + 后端正式字段保留”。

也就是说：

- 本地真机调试：优先用手机本地临时路径显示头像，解决不显示问题；
- 后端数据库：仍保存正式 `fileId`；
- 正式上线：应切换到 HTTPS 文件域名、OSS 或 CDN，通过正式图片地址显示。

当前方案不是最终文件存储方案，后续上线前仍需完成：

- 文件存储接入 OSS；
- 小程序合法域名配置；
- HTTPS 图片预览地址；
- 图片访问权限策略；
- 头像、证件、客服文件的统一存储规范。

## 六、责任边界分析

### Claude 侧问题

Claude 多次实现时更偏向代码结构和静态验证，没有充分覆盖真机调试行为。

主要不足：

- marker 验证通过不代表真机体验通过；
- 没有主动加入运行期诊断；
- 没有把“保存失败”和“图片显示失败”拆开验证；
- 对微信小程序 `<image>` 加载本地 HTTP 资源的风险判断不足。

### Codex 侧问题

Codex 前几轮复审也存在不足。

主要不足：

- 一开始也偏向审查字段保存和返回结构；
- 没有第一时间用接口冒烟测试确认后端链路；
- 没有尽早要求输出前端 `image.error` 和最终 `src`；
- 多轮修复后才把问题定位到本地真机图片预览链路。

### 脚本本身的问题

命令脚本不是直接根因。

脚本的问题在于验证范围不够：

- 能验证 build；
- 能验证 JS 语法；
- 能验证部分 marker；
- 不能验证微信真机 `<image>` 真实加载行为。

因此以后脚本不能替代人工真机验收，尤其是：

- 文件上传；
- 图片/视频预览；
- 微信授权；
- 手机号；
- 客服聊天；
- 地图定位；
- 支付。

## 七、以后类似问题的排查规范

遇到“上传成功但页面不显示”类问题，必须按下面顺序排查：

1. 确认上传接口是否返回稳定 `fileId`。
2. 确认保存接口请求体是否带了该 `fileId`。
3. 确认保存接口返回是否包含该字段。
4. 确认重新 GET 详情接口是否还能拿到该字段。
5. 直接请求图片预览接口，确认返回 200 和正确 `Content-Type`。
6. 在小程序页面打印最终 `<image src>`。
7. 监听 `<image binderror>`，确认是否是图片组件加载失败。
8. 真机调试和开发者工具模拟器都要测。
9. 如果本地 HTTP 不稳定，开发期可用本地临时路径兜底，正式环境必须用 HTTPS/OSS。

## 八、后续待办

1. 头像稳定后，删除或降级 `[AvatarDebug]` 诊断日志。
2. 设计正式文件存储方案：
   - 头像：public；
   - 简历公开图片：脱敏后 public；
   - 证件资料：private；
   - 客服图片/视频/文件：按会话权限控制；
   - 合同/PDF：private 或 signed URL。
3. 上线前接入 HTTPS 域名和 OSS/CDN。
4. 增加一个专门的“上传文件后可预览”接口测试脚本。
5. 对微信真机依赖能力建立人工验收清单，不能只依赖 Claude/Codex 静态验证。
