# 家政服务人员入驻小程序开发技术方案

版本：v0.1  
依据：`家政服务人员入驻_需求文档（独立版）.docx` v1.2  
范围：阶段一入驻 MVP。预约分销系统和《大家评评理》只做数据模型与接口预留，不实现完整业务。

## 1. 技术目标

本系统先做成独立的“服务人员资质中心”，核心目标是：

- 服务人员可以完成注册、资料填写、证件上传、提交审核、查看审核状态。
- 后台可以完成审核、驳回、要求补充、暂停/恢复、服务记录摘要维护、日志审计。
- 系统沉淀稳定 `staff_id`，后续预约分销系统、资金系统、《大家评评理》都用它对接。
- 本期不做 C 端下单、派单、支付、保证金、真实订单系统、真实评理流程。

## 2. 推荐技术栈

### 2.1 小程序端

- 微信原生小程序。
- TypeScript 可选；如果团队熟悉 JS，可先用 JavaScript。
- 组件库可选：TDesign Weixin 或 Vant Weapp。
- 文件上传通过后端接口中转，不直连 OSS。

### 2.2 后端

推荐：

- Node.js LTS
- NestJS
- Prisma ORM
- PostgreSQL
- JWT + 微信登录态
- Zod / class-validator 做接口校验

备选：

- Fastify + Prisma，开发更轻，但后台权限、模块划分需要自建更多约束。

### 2.3 后台管理端

推荐：

- React + Ant Design Pro

备选：

- Vue 3 + Element Plus

阶段一后台以表单、列表、审核流为主，React Ant Design 或 Vue Element Plus 都适合。

### 2.4 文件存储

- 开发环境：本地私有目录或 MinIO。
- 生产环境：阿里云 OSS 私有 Bucket。
- 小程序上传文件到后端，后端保存到 OSS。
- 后台查看证件时，由后端生成短时有效 URL 或后端代理读取。

## 3. 系统架构

```text
微信小程序
  -> HTTPS API
  -> 后端服务
  -> PostgreSQL
  -> OSS / 本地私有文件存储

后台管理端
  -> HTTPS API
  -> 后端服务
  -> PostgreSQL
  -> OSS / 本地私有文件存储
```

建议 API 路径分组：

- `/api/app/*`：小程序端接口。
- `/api/admin/*`：后台管理接口。
- `/api/integration/*`：后续预约分销系统、评理模块预留接口。

## 8. 状态机

### 8.1 入驻审核状态

```text
draft
  -> under_review
  -> need_more_info -> under_review
  -> rejected -> under_review
  -> approved
```

状态说明：

| 状态 | 说明 |
|---|---|
| `draft` | 草稿，未提交。 |
| `under_review` | 已提交，等待审核。 |
| `need_more_info` | 待补充资料。 |
| `rejected` | 审核驳回，可重新提交。 |
| `approved` | 审核通过。 |

规则：

- 资料未完整不可提交。
- 提交后关键资料和证件进入审核态。
- 待补充和驳回必须有原因。
- 审核通过后进入可服务状态判断。

### 8.2 单证件状态

```text
not_submitted
  -> under_review
  -> approved
  -> expired

under_review
  -> rejected
  -> under_review
```

规则：

- 证件过期默认只打标签、提醒、后台筛选。
- MVP 不因证件过期自动停派。
- 运营可以人工暂停。
- 新证件驳回时，旧证件仍有效则保留当前有效版本。

### 8.3 可服务状态

对外统一字段：

- `listing_status`: `not_listed / available / unavailable`
- `is_available`: `true / false`

内部原因字段：

- `pause_reason`: `none / manual_pause / external_freeze / admin_down`
- `credential_badge`: `normal / expired / missing / rejected`

```text
not_listed
  -> available
  -> unavailable
  -> available
```

触发规则：

- 审核通过后，如果没有后台暂停和外部冻结，可以进入 `available`。
- 后台手动暂停，进入 `unavailable`。
- 外部系统冻结，进入 `unavailable`。
- 证件过期默认不改变 `available`，只更新证件徽标和提醒。

### 8.4 服务记录摘要状态

```text
active
  -> voided
```

规则：

- 人工记录不物理删除。
- 修改、作废必须记录日志。
- 人工评理结果不触发资金、冻结、真实评理流程。

### 9.3 敏感字段控制

- 手机号

- 证件编号
- 出生日期
- 证件图片

- 必须有权限。
- 必须记录 `operation_log`。
- 建议后台要求填写查看原因。

## 10. 文件存储方案

### 10.1 文件类型

| 文件类型 | 存储权限 | 说明 |
|---|---|---|
| 头像 | 私有或受控公开 | 可展示，但上传和修改需记录。 |
| 身份证 | 私有 | 高敏感。 |
| 健康证 | 私有 | 敏感证件。 |
| 无犯罪证明 | 私有 | 高敏感。 |
| 保险凭证 | 私有 | 敏感证件。 |
| 技能证书 | 私有 | 可后续展示脱敏摘要。 |

### 10.2 对象路径

建议：

```text
staff/{staff_id}/avatar/{file_id}.{ext}
staff/{staff_id}/credential/{credential_type}/{version}/{file_id}.{ext}
```

### 10.3 上传流程

```text
小程序选择文件
  -> 上传到后端 /api/app/files/upload
  -> 后端校验类型、大小、权限
  -> 后端写入 OSS / 本地私有目录
  -> 写入 file_asset
  -> 返回 file_id
```

### 10.4 预览流程

```text
后台请求预览
  -> 校验权限
  -> 写 operation_log
  -> 生成短时 URL 或后端代理文件流
  -> 返回给后台
```

安全要求：

- OSS Bucket 私有。
- 不在小程序端保存 OSS AccessKey。
- 预览链接有效期建议 1 到 5 分钟。
- 后台文件预览必须留痕。

## 14. 主要风险与控制

| 风险 | 控制方式 |
|---|---|
| 把服务记录摘要误做成订单系统 | 明确字段来源，人工记录不触发支付、派单、资金。 |
| 证件图片泄露 | 私有存储、短时 URL、敏感查看日志。 |
| 状态机复杂化 | 对外统一 `is_available / listing_status`，内部用原因字段。 |
| 外部接口提前膨胀 | P1/P2 只预留表和接口契约，不进 P0。 |
| Claude Code 一次性实现过多 | 按阶段开发，每阶段验收后再继续。 |
